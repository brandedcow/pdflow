# PDF Extraction Backend — Design Spec
**Date:** 2026-05-10
**Status:** Approved

## Overview

A Python FastAPI service that accepts a PDF file, extracts structured text and tables using Marker, verifies the output with Groq's llama-3.3-70b, and returns a confidence-scored JSON response. The service runs locally during development. It is the source of truth for the shared API contract between backend and mobile.

---

## Stack

| Concern | Choice |
|---|---|
| Framework | FastAPI |
| PDF extraction | Marker |
| LLM verification | Groq (llama-3.3-70b) via `groq` Python SDK |
| Data validation | Pydantic v2 |
| Environment | `python-dotenv` — loads `GROQ_API_KEY` from shared `.env.local` at project root |
| Testing | pytest + `httpx` TestClient |
| API schema | OpenAPI (auto-generated by FastAPI at `/openapi.json`) |

## API Contract

### `POST /extract`

**Request:**
```
Content-Type: multipart/form-data
Body: pdf_file (binary)
```

**Response 200:**
```json
{
  "book_id": "uuid-string",
  "status": "success" | "partial" | "failed",
  "overall_confidence": 0.0,
  "page_count": 0,
  "blocks": [
    {
      "type": "heading" | "text" | "table",
      "content": "string",
      "page": 1,
      "confidence": 0.0
    }
  ]
}
```

**Response 422:** Pydantic validation error (malformed request)
**Response 500:** Unexpected server error

**Status rules:**
- `success` — `overall_confidence >= 0.8`
- `partial` — `overall_confidence >= 0.5` and `< 0.8`
- `failed` — `overall_confidence < 0.5` or any unrecoverable pipeline error

**Content field:**
- `heading` and `text` blocks: plain text
- `table` blocks: GitHub-flavoured Markdown table string

## Pydantic Models (`models.py`)

```python
from enum import Enum
from pydantic import BaseModel

class BlockType(str, Enum):
    heading = "heading"
    text = "text"
    table = "table"

class ExtractionStatus(str, Enum):
    success = "success"
    partial = "partial"
    failed = "failed"

class Block(BaseModel):
    type: BlockType
    content: str
    page: int
    confidence: float

class ExtractionResponse(BaseModel):
    book_id: str
    status: ExtractionStatus
    overall_confidence: float
    page_count: int
    blocks: list[Block]
```

## Pipeline

### `extractor.py`

Accepts a file path, runs Marker to produce structured output. Maps Marker's block types to `BlockType`. Returns `list[Block]` with `confidence` set to `0.0` (not yet scored) and `page` populated from Marker's output.

Marker is configured to extract text and tables only — equations and figures are excluded from v1 scope.

### `verifier.py`

Accepts `list[Block]` and the extracted Markdown context. Sends blocks in batches to Groq (to stay within context limits). Prompt instructs the model to:
1. Check each block's text for coherence and accuracy against source context
2. Return a confidence score (0.0–1.0) per block

Returns the same list with `confidence` populated. If Groq returns an error for a batch, that batch's blocks default to `confidence = 0.5` (partial, not failed) so a single API hiccup doesn't discard the whole document.

### `main.py`

1. On startup: validates `GROQ_API_KEY` is present — raises `RuntimeError` and exits if missing
2. `POST /extract`:
   - Generates `book_id` UUID
   - Saves uploaded file to a temp directory
   - Calls `extractor.extract(file_path)` → raw blocks
   - Calls `verifier.verify(blocks, context)` → scored blocks
   - Computes `overall_confidence` as mean of all block confidence scores
   - Determines `status` from thresholds
   - Returns `ExtractionResponse`
   - Cleans up temp file

## Environment

`.env.local` at the project root (shared with mobile app):
```
GROQ_API_KEY=your_key_here
```

`main.py` loads it with:
```python
from dotenv import load_dotenv
load_dotenv('../.env.local')
```

The server must be started from the `backend/` directory so the relative path resolves correctly:
```bash
cd backend && uvicorn main:app --reload
```

## Error Handling

| Failure | Behaviour |
|---|---|
| `GROQ_API_KEY` missing at startup | `RuntimeError` — server exits immediately with clear message |
| Marker fails to parse PDF | Return `status: failed`, `blocks: []`, `overall_confidence: 0.0` |
| Groq API error on a batch | Affected blocks default to `confidence: 0.5`, pipeline continues |
| Groq API error on all batches | Return `status: partial` with unscored blocks |
| Unexpected exception in route | Return HTTP 500 |

## Testing

**Strategy:** Unit-test `extractor.py` and `verifier.py` in isolation with fixture PDFs and mocked Groq. Integration-test `POST /extract` end-to-end with `TestClient` and a real academic paper fixture.

**`test_extractor.py`:**
- Fixture: a simple single-column PDF
- Fixture: a two-column academic paper PDF
- Asserts blocks have correct types, non-empty content, and valid page numbers

**`test_verifier.py`:**
- Mock Groq SDK to return known confidence scores
- Assert blocks are returned with correct confidence values
- Assert Groq error on one batch defaults those blocks to `0.5` without failing the rest

**`test_main.py`:**
- `POST /extract` with a valid PDF → asserts `200`, correct response shape
- `POST /extract` with a non-PDF file → asserts `422` or `500`
- `POST /extract` with Groq mocked to fail all batches → asserts `status: partial`

## Out of Scope (v1)

- Authentication / API keys between mobile and backend
- Cloud hosting
- Equation rendering (LaTeX / MathML)
- Figure / image extraction
- Multi-language document support
- Caching on the backend (mobile stores the result; backend is stateless)
